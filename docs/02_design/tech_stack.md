# 技術選定

## 技術選定ポリシー

1. **DDD準拠**: コアドメイン以外は原則として既存ライブラリ・既存サービスを利用する
2. **無料枠優先**: 収益化前提がない段階では、無料枠内で継続運用できる選択を優先する
3. **TDD適合性**: テストの自動化・モック化・CI組み込みがしやすい技術を選ぶ
4. **OOUI適合性**: オブジェクト中心の画面遷移を実装しやすいUI基盤を選ぶ

## 選定サマリー

| レイヤー | 採用技術 | 候補 | 選定理由 |
|---|---|---|---|
| モバイルアプリ | Flutter | React Native | 単一コードベースでiOS/Android対応、高パフォーマンス |
| バックエンド | Go (Gin) | Node.js, Ruby on Rails | 将来のバックアップ・共有API向けに軽量・高パフォーマンス |
| データベース | PostgreSQL | MySQL, SQLite | 信頼性、UUID対応、JSON型サポート |
| ストレージ | Cloudflare R2 | AWS S3 | 無料枠が大きく、個人開発向けコスト最適 |
| 認証 | 自前JWT + OAuth2 | Firebase Auth | 外部依存を最小化、将来のE2E暗号化に対応しやすい |
| インフラ | AWS (ECS Fargate / RDS / CloudWatch) | GCP, Azure | 学習コストが低く、将来の拡張性も高い。無料枠・低コスト構成から始められる |
| IaC | Terraform | AWS CDK, CloudFormation | AWSリソースを宣言的に管理し、構成差分と再現性を担保できる |
| CI/CD | GitHub Actions | CircleCI | 無料枠あり、GitHubとの親和性 |
| テスト | Go test / testify / Flutter test | - | TDDでの小さな反復を回しやすく、CIに統合しやすい |

## 詳細

### Flutter
- iOS / Android の両方に対応できる
- `sqflite` によるローカルDB（オフラインファースト対応）
- `flutter_secure_storage` でアプリロック情報や将来のトークンを安全に保存
- OOUIの「オブジェクト選択→操作」を画面構成に落としやすい

### Go (Gin)
- シングルバイナリでデプロイが簡単
- バックアップやエクスポートなどの非同期処理と相性が良い
- `sqlc` によるタイプセーフなSQLクエリ生成を採用
- Domain/Application/Infrastructure の分離を実装しやすい

### PostgreSQL
- v1.x 以降のバックアップ・共有データ保管先として扱う
- `recorded_at` と `created_at` の分離クエリが得意
- 将来的な全文検索（`pg_trgm`）拡張に対応

### 永続化先として RDS (PostgreSQL) を選んだ背景

- 本プロダクトの主要クエリは「日付範囲 + 記録対象 + タグ」の組み合わせ検索であり、RDBのほうが素直に表現できる
- `entries` と `entry_tags`、`entry_media` をまたぐ整合性をトランザクションで担保しやすい
- エクスポートや将来の月次レポートで集計クエリが増える想定があり、SQLの運用学習コストが低い
- DDDの観点で、当面はドメイン実装に集中したいため、アクセスパターン最適化に時間がかかるNoSQL設計を避ける

### DynamoDB 再検討（コスト観点）

#### 結論
- **現時点では主永続化は RDS を維持**する
- ただし、将来的にアクセスパターンが単純化し、SQL集計要件が弱まる場合は DynamoDB を再評価する

#### 再検討結果サマリー

| 観点 | RDS (Aurora Serverless v2) | DynamoDB (On-Demand) |
|---|---|---|
| 低トラフィック時コスト | 低いが最低コストは発生しやすい | 非常に低く抑えやすい |
| クエリ柔軟性 | 高い（複合条件、集計、JOIN） | 低い（アクセスパターン先決めが前提） |
| 開発学習コスト | 既存知識を使いやすい | GSI設計やパーティション設計の学習が必要 |
| 要件適合 | 高い | 中（要件をNoSQL向けに再設計する必要あり） |

#### DynamoDB を採用する条件

- APIのアクセスパターンを固定し、JOIN不要の単純な読み書き中心に再設計できる
- 月次レポートや複合検索を別基盤（Athena等）に分離する方針が固まる
- コストが主要課題となり、再設計コストを上回る削減効果が見込める

### Cloudflare R2
- S3互換APIで将来の移行が容易
- 外部転送料金が無料（家族利用規模では実質無料）

### Terraform
- AWSインフラ（VPC、ALB、ECS、RDS、CloudWatch、IAM）をコードで管理する
- `plan` をPRレビューに含め、`apply` は本番承認後に実行する
- 環境差分は `development` / `staging` / `production` の変数管理で吸収する

## セキュリティ考慮

- パスワードハッシュ: `bcrypt` (cost=12)
- JWT有効期限: アクセストークン15分、リフレッシュトークン30日
- レートリミット: ログイン試行は10回/分まで
- SQLインジェクション対策: プレースホルダーによるパラメータバインド（`sqlc`）
- メディアURL: 署名付きURL（有効期限付き）でダイレクトアクセスを防止

## TDD運用方針

- 実装前に失敗するテストを先に作成し、最小実装でテストを通す
- 1機能につき「ドメインロジックテスト」と「ユースケーステスト」を最低1本ずつ作成する
- バグ修正時は再発防止テストを先に追加してから修正する
- CIではテストを必須ゲートとし、失敗時はマージしない
