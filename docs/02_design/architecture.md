## アーキテクチャの原則

1. **ローカルファースト**: v0.1 ではネットワークなしでも記録・閲覧・保護が完結する
2. **入力の軽さを最優先**: 家事や育児の合間でも短時間で記録できる導線を維持する
3. **プライバシーを初期設計に組み込む**: 端末内保護、暗号化、エクスポート性を最初から考慮する
4. **クラウド連携は後付け可能にする**: 将来の共有やバックアップを追加してもローカル中心設計を崩さない

## 設計方針

### UI設計（OOUI）

- 基本動線は **「オブジェクトの選択」→「操作」** とする
- 主オブジェクトは `記録対象 (Subject)`、`記録 (Entry)`、`タグ (Tag)`
- 画面遷移は「誰の記録か」を先に決め、その後に作成・編集・検索・振り返りを行う
- 一覧からオブジェクトを選択した時点で、実行可能な操作のみを提示する

### ドメイン設計（DDD）

- コアドメインは「家族のリーダーが記録を残し、見返して気遣いにつなげる体験」
- 自前実装はコアドメインに集中し、非コア領域は既存ライブラリや外部サービスを優先する
- レイヤーは `Domain` / `Application` / `Infrastructure` を明確に分離する
- 外部サービス選定は、現時点で収益化を見込まない前提で無料枠を優先する

## コンポーネント責務

| コンポーネント | 責務 |
|---|---|
| モバイルアプリ | UI/UX、ローカルDB、アプリロック、記録検索、振り返り表示 |
| ローカルストレージ | エントリ・タグ・記録対象・設定などの永続化 |
| APIサーバー | 将来のバックアップ、共有、エクスポート要求の受付 |
| PostgreSQL | 将来のアカウント・同期・共有用の構造化データ |
| S3互換ストレージ | 写真・音声などのバイナリデータ |

## 採用パターン

- **Repository Pattern**: データアクセス層を抽象化し、テスタビリティを確保
- **ローカル優先同期**: 将来的にクラウド連携を行う場合も、ローカルを正として後同期する
- **イベント駆動**: 振り返り通知やエクスポートなどの非同期処理はジョブキューを利用
- **TDD駆動実装**: 先にテストで期待振る舞いを定義し、回帰を防ぎながら実装を進める
